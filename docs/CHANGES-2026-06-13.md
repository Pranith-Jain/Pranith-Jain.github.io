# Live-IOC Hardening & Edge-Cache Notes — June 13, 2026

> Documents the root-cause of the "24 of 36 sources show 0" state observed
> before the deploy, the four-part fix shipped in `88027d8c`, and the edge
> cache behavior to expect going forward. Read alongside `worker/scheduled.ts`
> (cache-warm cron) and `api/src/routes/live-iocs.ts` (synchronous fan-out).

## Background

The Live IOC page renders the union of items contributed by every
"registered" feed source (currently 35). The backend serves a single
capped Cache API response (1h TTL) plus a per-colo Durable Object slice
filled by the `FEEDS_QUEUE` consumer (one message per source, 3-min
timeout, 3 retries). On a cold start or slice miss, a fallback
`synchronous fan-out` walks all 35 sources with `concurrentMap` and fills
the cache.

The pre-deploy symptom: the page header reported `"Sources: 24/36"`
sources contributing items, with the other 12 showing `ok:false` or
`count:0`. Operators could not tell whether the upstream feeds were sick
or whether the platform was masking real data.

## Root cause

A free-plan Worker invocation has a **50-subrequest cap**. The cache-warm
cron (`worker/scheduled.ts`) fans out 22 endpoints with `Promise.allSettled`
every hour, with most endpoints already consuming 1-3 subrequests each.
Adding the live-iocs synchronous fan-out (35 sources × 1 fetch each =
35 subrequests) on top exhausted the budget. Result: every source beyond
the budget was reported as `ok:false count:0`, the 1h cache absorbed the
degraded snapshot, and the same poisoned state was served until the
next cache-warm tick.

## Fix (`88027d8c`)

### 1. Subrequest budget guard — `api/src/routes/live-iocs.ts`
- Lowered `FEED_FANOUT_CONCURRENCY` from 8 → 5.
- Added a shared subrequest budget (`max 30`) on `FeedDeps`. The
  `fetchLiveIocs` wrapper short-circuits a source to a `ok:false` stub
  when the budget is exhausted — we degrade gracefully (a few sources
  miss this snapshot) instead of throwing and dropping ALL 35 sources.
- The compose-on-read path stays the primary read model; the guard
  only matters on a cold start / slice miss.

### 2. Removed `/api/v1/live-iocs` from cache-warm — `worker/scheduled.ts`
- The cache-warm cron now fans out 21 endpoints (was 22, with
  `live-iocs` excluded).
- Live-IOCs warms itself on the first real page visit via single-flight.
- The hourly cron already enqueues 35 per-source queue messages that
  the consumer fills into the per-colo Cache API slices, so the
  user-facing read path never depends on cache-warm.

### 3. Fallback URLs for text-feed sources — `api/src/routes/live-iocs.ts`
- `textFeedSource` supports an optional `fallbackUrls` list. When the
  primary URL fails, the wrapper tries each fallback in order before
  giving up.
- Wired for `binarydefense` (own-site mirror) and `tor-exit` (canonical
  Tor Project exit list). The two CPS_BASE feeds with well-known
  alternates.

### 4. Per-feed health summary — `src/pages/threatintel/LiveIocs.tsx`
- The header now shows a compact
  `feed health: N healthy · M empty · K unreachable` line under the
  "Sources: ..." prose, with the unreachable IDs in a tooltip.
- Derives purely from the existing `registered_sources` field; the
  unreachable count is highlighted in rose.
- When all 35 feeds are healthy the line shows `0 unreachable` in
  neutral slate (no false alarms).

## Diagnostic endpoint — `?debug=1`

Added a non-hot-path escape hatch:

```
GET /api/v1/live-iocs?debug=1
```

Bypasses the cache and runs a fresh synchronous fan-out with per-source
error diagnostics. The response body is augmented with a `debug` object
describing every registered source's actual HTTP outcome:

```json
{
  "debug": {
    "duration_ms": 3082,
    "total": 35,
    "ok": 32,
    "failing": ["malwarebazaar", "phishing", "mythreatintel"],
    "entries": [
      { "id": "tweetfeed", "ok": true,
        "attempts": [{ "url": "...", "ok": true, "status": 200, "bytes": 2584 }] },
      ...
    ]
  }
}
```

`?debug=1` cost: a 35-subrequest synchronous fan-out (~3-12s wall time,
50-subrequest cap respected). Do not link this from the public page.

## Edge-cache behavior — what to expect

| Read path | Source | Latency | Cache lifetime |
|---|---|---|---|
| `GET /api/v1/live-iocs` (warm) | `c.env.CACHE` (per-colo Cache API slice) | 50-200ms | 1h, replaced on first read after consumer fill |
| `GET /api/v1/live-iocs` (cold) | DO consumer + `concurrentMap` fallback | 3-12s | 1h (degraded → graceful stub) |
| `GET /api/v1/live-iocs?debug=1` | direct fan-out, no cache | 3-12s | n/a |
| `GET /api/v1/live-iocs` (per-source slice miss) | `concurrentMap` with budget guard | <8s | 1h per slice |

The page itself (`/threatintel/live-iocs`) is **prerendered HTML** with
the chrome + skeleton from `dist/__prerendered/threatintel__live-iocs.html`.
The dynamic header (sources sentence + feed-health line + IOC list) is
filled in by `useEffect` after the JS bundle hydrates. The "feed health"
line you see in the UI is always sourced from the latest API call, not
the prerender.

## Hourly cron (cache-warm)

```
schedule: 0 * * * *
```

Fans out 21 endpoints via `Promise.allSettled`. Live-IOCs is
**deliberately excluded** (see Fix #2). The consumer (max_batch_size=1)
fills per-source KV slices with `live-iocs:slice:<id>`, which the
read-side compose function assembles on the next page hit.

## What I intentionally did NOT change

- **Do not directly purge the edge cache** to chase the pre-deploy
  artifact. The 1h TTL will absorb the next good snapshot within the
  hour, and the per-source slices in KV are already being refreshed
  by the hourly consumer.
- **Do not re-add live-iocs to the cache-warm list** without also
  cutting the other 21 endpoints. The 50-subrequest cap is a free-plan
  constraint, not a tuning knob.
- **Do not raise `FEED_FANOUT_CONCURRENCY` past 5** without first
  measuring end-to-end subrequest count on a cold start. 5 keeps us
  inside the 30-source budget guard even with the diagnostic mirror
  map loaded.

## 2026-06-15 — bg-fan-out attempt + revert

Attempted to move the sync fan-out into `ctx.waitUntil` so the page handler
could serve the composed-from-slices response synchronously while a
background fan-out repopulated the response cache. Deployed briefly,
reverted within an hour. Why:

**It didn't work — `waitUntil` shares the 50-subrequest cap.** The bg
fan-out's `fetchLiveIocs` threw `Too many subrequests by single Worker
invocation` (verified in the wrangler tail), so the `cache.put` never
ran and the response cache stayed stale. The user-visible regression
was even worse than the original problem: the new "true cold start"
branch returned `{ total: 0, items: [], sources: [], registered_sources: [] }`
and cached that empty shape for 5 min, so the page showed
*"No indicators in the current snapshot. The cron repopulates this
every 15 minutes"* for that whole window.

The architectural lesson I missed: **`waitUntil` runs in the same
Worker invocation and shares the subrequest cap.** It's not a free
separate budget. A long-running bg task that uses 30+ subrequests
will still trip the cap, just deferred to after the response is sent.

**Reverted to the 88027d8c baseline** — the user now sees
`10 healthy · 2 empty · 24 unreachable` again, which is the same
state they had before this experiment. The 88027d8c budget guard
prevents the cliff from "0 healthy" but can't push past ~10 healthy
because the sync fan-out itself is too big for the 50 cap on a
36-source registry.

The honest floor for the current architecture on the free plan:
**~10-12 healthy out of 36 sources**, with the rest short-circuiting
or being cancelled by the Worker-level cap. The 22-25 unreachable
are not real upstream failures — `?debug=1` confirms the upstreams
return 200 from the same Worker. They're the budget math.

## Real fixes (deferred — each is a substantive refactor)

1. **Move the sync fan-out out of the request path entirely.** The
   `waitUntil` cap-sharing issue is fundamental. The right move is to
   never run the sync fan-out from a page hit at all. On a true cold
   start, return an empty/degraded response and let the queue
   consumer fill the slices (it already runs with `max_batch_size: 1`
   so each source gets its own 50-cap). Trade-off: the user sees an
   empty page for one request on a fresh colo, then full data.

2. **Chunked fan-out across multiple Worker invocations.** A
   `ctx.waitUntil` that does `env.SELF.fetch('/api/v1/_internal/sync-15-sources')`
   to start a separate Worker invocation (with its own 50-cap) for
   each chunk of 15 sources. Each chunk completes in its own budget.
   2-3 chunks = full 36 sources covered. The internal route would
   need to be wired but each chunk's wall-time is independent.

3. **Pay for the Workers Paid plan (1000 subrequest cap).** Not a code
   change. Removes the ceiling entirely. Costs $5/mo as of 2026 pricing.

4. **Skip the sync fan-out when slices exist (even if degraded).** Change
   `if (presentSlices > 0 && response.total > 0) return response;` to
   `if (presentSlices > 0) return response;`. Trade-off: the user
   sees the slice data (which the consumer wrote), and the consumer
   becomes the only path that determines healthy/unhealthy status.
   If the consumer is healthy, the user sees 32/35. If the consumer
   is broken, the user sees 0/35 (worse than 10/2/24) until the
   consumer recovers. This is the right move IF we add health
   monitoring to the consumer so we know it's working.

**Recommended:** (1) + (4) together. The consumer is already
correct (each source has its own 50-cap); the page-hit path should
trust it. If the consumer fails, the operator sees it via the
`job: live-iocs-slice` logs and can intervene.
